# 11.15 — Playwright Testing

[Playwright](../glossary.md#playwright) is a tool that drives a real browser to test your app end to end — it clicks buttons and checks results the way a user would. This guide uses AI sessions to generate those tests from your project docs; the session steps follow [11.03 — Run an AI session](11.03-run-ai-session.md).

**Time:** 1–3 hrs per module  
**Owner:** Developer / QA engineer  
**Prerequisites:** Phase 6 implementation complete for the module; [`{MODULE}-QA-*` tickets](../glossary.md#qa-ticket-ids) defined in module breakdown; Playwright installed in the code repo — check with `npx playwright --version`; if missing, install with `npm init playwright@latest` or your team's setup doc

---

## Exit gate

All `{MODULE}-QA-*` tickets for the sprint have corresponding passing Playwright tests.  
`TEST-CONTEXT-{MODULE}.md` committed to context repo.  
`tests/e2e/{module}.spec.ts` committed to code repo.

---

## Steps

| Step | Action | Output | Verification |
|------|--------|--------|--------------|
| 1 | Create test context doc | `docs/06-development/TEST-CONTEXT-{MODULE}.md` in context repo | All QA tickets for the module are listed; environment details filled in |
| 2 | Generate Playwright script via AI session | `tests/e2e/{module}.spec.ts` in code repo | Every scenario in test context has a matching `test()` block; no placeholder steps |
| 3 | Run Playwright and triage results | Pass/fail report | All tests green; failures filed as new `{MODULE}-QA-*` tickets |

---

## Step 1 — Create test context doc

**Goal:** Produce a structured doc the AI can load in Step 2 to generate accurate, grounded Playwright tests — not hallucinated selectors.

1. Open a new AI session ([11.03 — Run an AI session](11.03-run-ai-session.md)). Load the [Tier-1 hot files](../glossary.md#tier-1-hot-files):
   - `AGENTS.md`
   - `PROJECT-INDEX.md`
   - `.ai/context/project-overview.md`
   - `.ai/AI-ASSISTANT-RULES.md`

2. Load module-specific context:
   - `docs/05-breakdown/modules/{NN}-{module}.md`
   - Any relevant screen/component names or selector hints

3. Use this AI prompt:

```markdown
Load the module breakdown for [{MODULE}] and all {MODULE}-QA-* tickets.
Draft docs/06-development/TEST-CONTEXT-{MODULE}.md using the TEST-CONTEXT template.
For each QA ticket:
- Map it to one or more test scenarios
- Describe the user flow step by step (no code yet)
- List the UI elements or API endpoints involved
- Note the expected outcome
Output the full markdown file.
```

4. Review the output:
   - Every `{MODULE}-QA-{NNN}` ticket appears in the scenario table
   - User flows are written in plain English (not Playwright code)
   - Environment section is filled in (base URL, auth method, seed data)

5. Commit `TEST-CONTEXT-{MODULE}.md` to the context repo:
   ```
   docs/06-development/TEST-CONTEXT-{MODULE}.md
   ```

---

## Step 2 — Generate Playwright script

**Goal:** Use the test context doc as AI input to produce a runnable, accurate Playwright spec file.

1. Start a **new AI session** (do not reuse the Step 1 session).

2. Load Tier-1 hot files + test context:
   - `AGENTS.md`, `PROJECT-INDEX.md`, `project-overview.md`, `AI-ASSISTANT-RULES.md`
   - `docs/06-development/TEST-CONTEXT-{MODULE}.md`
   - Any existing Playwright config or helper files in `tests/e2e/`

3. Use this AI prompt:

```markdown
Load TEST-CONTEXT-{MODULE}.md. Generate tests/e2e/{module}.spec.ts as a Playwright TypeScript
test file. Requirements:
- One test() block per scenario in the test context
- Use the scenario ID ({MODULE}-QA-{NNN}) as the test description prefix
- Use real selectors from the app (data-testid preferred; fall back to role/label)
- Include beforeEach for auth/setup if needed
- No placeholder comments like "// TODO: add assertion"
- Follow the Playwright patterns in any existing tests/e2e/*.spec.ts files
Output the complete file.
```

4. Review the generated file:
   - Every scenario from TEST-CONTEXT is covered
   - Selectors reference actual elements (verify against the running app if needed)
   - Assertions are meaningful — not just `expect(true).toBe(true)`
   - Auth/setup aligns with the environment section of TEST-CONTEXT

5. Commit `tests/e2e/{module}.spec.ts` to the **code repo** (not the context repo).

---

## Step 3 — Run Playwright and triage results

1. Run the tests:
   ```bash
   npx playwright test tests/e2e/{module}.spec.ts
   ```
   Or with the project-specific command from `package.json`.

2. On **all tests pass:**
   - Mark all `{MODULE}-QA-*` tickets as done in the sprint board
   - Add a `tests/e2e/` entry to the PR description

3. On **test failures:**

   | Failure type | Action |
   |--------------|--------|
   | Selector not found | Fix selector in spec file; re-run |
   | Assertion mismatch (expected behavior) | The feature has a bug — file a new `{MODULE}-QA-{NNN}` defect ticket |
   | Assertion mismatch (wrong expectation) | Update the test context doc and re-generate the assertion |
   | Auth / setup failure | Fix `beforeEach`; check test environment section of TEST-CONTEXT |
   | Timeout | Check network/API stubs; increase timeout only as last resort |

4. Repeat Step 3 until all targeted tickets are green.

5. On repeated failures that indicate a spec or implementation gap, start a new AI session:

```markdown
Load TEST-CONTEXT-{MODULE}.md and the Playwright output below.
Identify why {MODULE}-QA-{NNN} is failing and suggest a fix.
[paste test output]
```

## Common mistakes

| Mistake | Consequence | Fix |
|---------|-------------|-----|
| Skipping Step 1 (no test context doc) | AI hallucinates selectors and flows | Always produce TEST-CONTEXT before generating the spec |
| Running tests before the feature is deployed | All tests fail; wasted triage time | Confirm the implementation PR is merged and deployed first |
| Storing `*.spec.ts` in the context repo | Wrong repo; breaks Playwright config | Spec files live in the **code repo** under `tests/e2e/` |
| Reusing the same AI session across all three steps | Context pollution; contradictory output | Start a fresh session for Step 2 |
| Not loading existing `spec.ts` patterns in Step 2 | Inconsistent style; duplicate helpers | Load at least one existing spec file so AI follows project conventions |
| Treating every timeout as a bug | Masks real flakiness | Investigate root cause before increasing timeouts |

| Step | Copy-paste opener |
|------|-------------------|
| 1 — Draft test context | `Load {MODULE} breakdown + QA tickets. Draft TEST-CONTEXT-{MODULE}.md. Map each QA ticket to a plain-English scenario with UI elements and expected outcome.` |

| 3 — Triage failure | `Load TEST-CONTEXT-{MODULE}.md. Explain why this Playwright failure occurs and suggest a targeted fix: [paste output]` |
