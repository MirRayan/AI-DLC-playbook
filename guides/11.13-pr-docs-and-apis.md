# Guide 11.13 — Open a PR That Touches Docs or APIs

> **Time:** 15–30 min | **Owner:** PR author | **Prerequisites:** Changes ready in context repo (and code repo if paired)

---

## Steps

| Step | Action | Verification |
|------|--------|--------------|
| 1 | Open `docs/process/PR-CHECKLIST.md` in your context repo | Checklist visible |
| 2 | **Light pass** (every docs PR): update `PROJECT-INDEX.md` dates/status if anything changed | Index reflects current state |
| 3 | **If API changed:** update `BACKEND-INDEX.md` + `docs/04-reference/` in **same PR** | Registry and contract match |
| 4 | **If architecture changed:** add/update ADR + `docs/03-architecture/decisions/README.md` | ADR index current |
| 5 | **If new decision:** update Pending decisions in index | Blockers visible |
| 6 | **If many links edited:** run `./scripts/check-doc-links.sh` | No broken internal links |
| 7 | Request review from **doc DRI** for quarterly-touching or index changes | Reviewer assigned |

---

## Light pass checklist (every docs PR)

From `docs/process/DOCUMENTATION-SYNC.md`:

- [ ] `PROJECT-INDEX.md` — Last Updated date if status changed
- [ ] Cross-links between changed docs are correct
- [ ] ADR README index updated if ADR added/changed
- [ ] No placeholder text (`{Project Name}`, `TODO`, `TBD`) left in changed files
- [ ] Module docs still reference valid `api:*` IDs

---

## API change checklist

- [ ] `BACKEND-INDEX.md` row added/updated
- [ ] `docs/04-reference/{module}-api.md` updated
- [ ] Module breakdown `api:*` refs still valid
- [ ] Status reflects reality (`verified` only after environment test)
- [ ] Deprecated endpoints marked `deprecated` with successor ID

---

## Pair code + docs PRs

When code changes API usage:

1. Code PR in application repo
2. Docs PR in context repo (same day, linked in PR description)
3. Both reference same ticket ID

Example PR description:

```
Implements HOME-003 hero banner integration.

Docs PR: [context-repo#42] — adds api:home:banners-hero to registry
```

---

## Deep pass (quarterly / pre-release)

Assign a DRI. Use `.ai/workflows/architecture-review-checklist.md`:

- Spec ↔ breakdown ↔ ADR alignment
- `project-overview.md` matches current reality
- API registry complete for release scope
- Pending decisions resolved or explicitly deferred
