# Guide 11.03 — Run Your First (or Any) AI Session

> **Time:** ~10 min to start | **Owner:** Any role | **Prerequisites:** Context repo with populated [hot files](../glossary.md#hot-file) (Phase 0 complete). Concepts first: [AI Assistant Basics](../00-ai-assistant-basics.md) explains what a session is and what "loading" means — read it if any of this page feels unfamiliar.

**Works with any AI tool.** In an [agentic tool](../glossary.md#agentic-tool) (e.g. Cursor, Claude Code) opened in your context repo folder, the prompts below work as written — the tool reads the files by path. In a plain chat tool (e.g. ChatGPT or Claude in the browser), "load" means you attach each file or paste its contents into the chat instead.

---

## What you will have when done

- AI loaded with correct project context
- Task-specific docs loaded
- Clear session goal stated
- Work proceeding in the correct repo (code vs context)

---

## Steps

| Step | Action | Verification |
|------|--------|--------------|
| 1 | Paste the **Tier-1 load prompt** (below) into your AI tool — "[Tier-1 hot files](../glossary.md#tier-1-hot-files)" are the four files loaded at the start of every session | AI lists correct project name, phase, architecture, constraints |
| 2 | If AI facts are wrong — **stop**. Fix hot files before coding | Facts match `PROJECT-INDEX.md` |
| 3 | Load **task context** — module doc, API ref, or ADR for today's work | AI references correct module/feature |
| 4 | **State intent** using the intent template (below) | AI restates goal accurately |
| 5 | If change touches schema/API/interfaces: run [impact analysis](../11-impact-analysis-before-change.md) **before** plan | Blast radius documented; approval received |
| 6 | For substantive work: ask for a **short plan**; approve before edits | Plan matches scope; no surprise file changes |
| 7 | Implement in **code repo** (not context repo unless docs change) | Code compiles / tests pass |
| 8 | If contracts or decisions changed: follow [11.13 — PR docs and APIs](11.13-pr-docs-and-apis.md) | PR checklist complete |

## Tier-1 load prompt (copy-paste)

Replace `[Project Name]` with your project. Plain-chat users: attach or paste the four files' contents where the prompt says to load them.

```markdown
I'm working on [Project Name].

Please load these files in order:
1. AGENTS.md
2. PROJECT-INDEX.md
3. .ai/context/project-overview.md
4. .ai/AI-ASSISTANT-RULES.md

Then confirm you understand:
- Project name and purpose
- Architecture approach
- Current phase
- Key constraints and ALWAYS/NEVER rules
```

---

## Task context load (Step 3)

Add to your prompt after Tier-1 succeeds:

```markdown
Also load for today's task:
- docs/05-breakdown/modules/{NN}-{module-name}.md
- docs/04-reference/{module}-api.md (if integrating APIs)
- docs/03-architecture/decisions/ADR-{NN}-*.md (if architecture-sensitive)
```

---

## Intent template (Step 4)

```markdown
Today I want to: [specific goal]

Expected outcome: [done when — e.g. "HOME-003 ticket complete, unit tests pass"]

Constraints:
- [e.g. follow MVVM pattern per ADR-002]
- [e.g. do not change API contracts without updating BACKEND-INDEX]
- [e.g. iOS + Android parity required]
```

## Clear session before a new task or feature

**Start a new chat** — do not continue a prior conversation when:

| Situation | Action |
|-----------|--------|
| Switching to a different module or feature | **New chat** |
| Switching code repo (e.g. iOS → Android) | **New chat** |
| Previous ticket completed; starting next ticket | **New chat** |
| AI references wrong task, files, or module | **New chat** |
| Same ticket, same day, continuing work | May continue — paste handoff summary |
| Long session (100+ messages) | Refresh context or **new chat** per `CONTEXT-RECOVERY.md` |

### New chat startup steps

1. Close or abandon the previous chat — do not rely on stale context
2. Optional: save one-line handoff to ticket or module doc ("HOME-002 done; HOME-003 next")
3. Open **new chat** → Tier-1 load only (not prior task files)
4. Load **only** this task's context (one module + relevant API/ADR)
5. State intent for **this task only**
6. If change touches schema/API → [impact analysis](../11-impact-analysis-before-change.md) before any edits

### New chat vs context refresh

| | New chat | Context refresh (same session) |
|---|----------|-------------------------------|
| **When** | New task, new feature, polluted context | Same ticket, 100+ messages, topic drift |
| **Action** | Fresh conversation | Reload hot files + task docs in place |
| **Loads** | Tier-1 + single task context | Re-verify understanding |

## Session types

| Type | Extra context to load | Example intent |
|------|----------------------|----------------|
| Feature development | Module breakdown + API ref | "Implement AUTH-005 OTP verification screen" |
| Bug fix | Relevant module + error context | "Fix crash on home banner when API returns empty" |
| Architecture / ADR | ADR template + related ADRs | "Draft ADR-003 for caching strategy" |
| Spec refinement | Concept note + specification | "Add FR-CHECKOUT-004 for guest checkout" |
| Doc sync | PR-CHECKLIST + changed files | "Update BACKEND-INDEX for new product list endpoint" |

---

## Common mistakes

| Mistake | Fix |
|---------|-----|
| Skip Tier-1 load | Always load hot files — every session |
| Reuse chat for unrelated next ticket | Start new chat per [Clear session](#clear-session-before-a-new-task-or-feature) |
| Code in context repo | Code changes go in application repo |
| AI wrong but proceed anyway | Fix `project-overview.md` or `AI-ASSISTANT-RULES.md` first |
| "Small" schema change without impact analysis | Run [impact analysis](../11-impact-analysis-before-change.md) first |
| Change API without registry update | Always pair code + `BACKEND-INDEX.md` + reference doc |
