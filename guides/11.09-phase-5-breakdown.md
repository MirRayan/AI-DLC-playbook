# Guide 11.09 — Phase 5: Breakdown

> **Time:** 2–5 days | **Owner:** Dev lead + PM | **Prerequisites:** [11.07 — Phase 3: business & tech context](11.07-phase-3-business-tech-context.md) (11.08 optional)

---

## Exit gate

The [exit gate](../glossary.md#exit-gate) for Phase 5: the first [sprint's](../glossary.md#sprint) [modules](../glossary.md#module) have implementable [tickets](../glossary.md#ticket) with IDs. A sprint plan exists. The [API registry](../glossary.md#backend-index) is started (if HTTP APIs). Module docs reference [`api:*` IDs](../glossary.md#api-registry-ids), not raw URLs.

---

## Steps

| Step | Action | Output | Verification |
|------|--------|--------|--------------|
| 1 | Create `docs/05-breakdown/SPRINT-PLAN.md` — sprints, milestones, module→sprint map | Plan exists | PM + dev lead agree |
| 2 | Copy module template → `docs/05-breakdown/modules/{NN}-{name}.md` per module | One file per module | Numbering aligns with spec |
| 3 | Per module: metadata table, layered tickets (Data/Domain/UI or equivalent), QA tickets | Ticket IDs assigned | Each ticket traceable to FR |
| 4 | **If HTTP APIs:** verify endpoints; add rows to `docs/05-breakdown/backend/BACKEND-INDEX.md` | Registry populated | Stable `api:*` IDs assigned |
| 5 | For verified APIs: create `docs/04-reference/{module}-api.md` from template | Contracts exist | Paths, payloads documented |
| 6 | Optional: create `docs/05-breakdown/sprints/S1/` with sprint-scoped ticket lists | Sprint scope clear | Matches SPRINT-PLAN |
| 7 | Optional: create `docs/05-breakdown/STORY-POINTS.md` linking to external tracker | Estimates documented | Jira/Linear keys if used |
| 8 | Update `PROJECT-INDEX.md` — module table, phase → 5 complete | Index current | Module count matches breakdown |

## Module template source

Use `templates/mobile-module-breakdown.md` in your [bootstrapped](../glossary.md#bootstrap) context repo — it was copied there from the bundled template when you ran [11.01 — Create a context repo](11.01-create-context-repo.md) (original: [AI-Driven-Project-Template/templates/mobile-module-breakdown.md](https://github.com/MirRayan/AI-DLC-playbook/blob/main/AI-Driven-Project-Template/templates/mobile-module-breakdown.md)).

Despite the "mobile" name, the template adapts to any project type: rename your copy to `module-breakdown.md` and drop the mobile-specific sections (iOS/Android platform rows) if you are not building mobile.

## AI assist prompt — module breakdown

```markdown
Load docs/02-specification/SPECIFICATION.md sections for [module name].
Load templates/mobile-module-breakdown.md (or docs/05-breakdown module template).

Produce docs/05-breakdown/modules/[NN]-[name].md with:
- Metadata table: sprint, spec refs, dependencies, milestone
- Data layer tickets: [MODULE]-001 format
- Domain layer tickets
- UI/Presentation tickets
- QA tickets from acceptance criteria
- Backend dependency section: api:* IDs only (not full URLs)

Platform value: Both | iOS | Android | Web | Backend as applicable.
```

## API registry — first rows

In `BACKEND-INDEX.md`:

| api:* ID | Method | Path | Status | Reference |
|----------|--------|------|--------|-----------|
| `api:auth:otp-send` | POST | `/V1/otp/send` | verified | `01-auth-api.md` |
| `api:product:list` | GET | `/V1/products` | pending | — |

**Status values:** `verified` | `pending` | `not_deployed` | `deprecated`

**Rule:** Never rename an `api:*` ID once published in module docs.

---

## Three-layer API documentation

| Layer | Location | Purpose |
|-------|----------|---------|
| Registry | `BACKEND-INDEX.md` | Readiness + stable IDs |
| Contract | `docs/04-reference/*.md` | REST paths, payloads for implementers |
| Deep context | `docs/05-breakdown/backend/{module}/` | Admin paths, edge cases, backend internals |

Module breakdown docs reference **`api:*` IDs only**.
